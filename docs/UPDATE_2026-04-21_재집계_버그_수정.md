# 재집계 동작 관련 두 건의 근본 원인 수정 (v1.1.1)

## 증상

1. 대시보드에서 "재집계" 버튼을 눌렀던 PC에서만 검은 CMD 창이 **반복적으로**
   튀어나왔다. 무의미해 보이는 숫자(XML 이벤트 레코드 ID·타임스탬프)가 빠르게
   스크롤되는 모습이 사용자에게 보였고, 재집계하지 않은 PC에서는 발생하지 않았다.
   한 번 뜨는 게 아니라 하트비트 주기(약 60초)마다 계속 재발했다.
2. 재집계를 수행해도 대시보드의 "마지막 종료" 시간이 갱신되지 않고
   부정확한 값(하트비트 마지막 수신 시각 근방)으로 고정되어 있었다.

## 근본 원인 (두 가지 결함의 조합)

### 결함 1: `wevtutil` 호출에 `CREATE_NO_WINDOW` 누락

Agent EXE는 PyInstaller `--windowed`로 빌드되어 부모 콘솔이 없다.
이 상태에서 `subprocess.run(['wevtutil', ...])` 호출이 `CREATE_NO_WINDOW`
없이 실행되면 Windows가 자식 프로세스용 **새 콘솔 창을 생성**한다.
`capture_output=True`로 stdout이 파이프로 리다이렉트되더라도 콘솔 창 자체는
생성되어 화면에 깜빡였다.

문제 위치 — `agent/installer.py`:
- `get_boot_events_from_log()` (EventID 6005/12 조회)
- `get_shutdown_events_from_log()` (EventID 6006/6008/1074 조회)

### 결함 2: 재집계 처리 멱등성 부재로 인한 무한 반복

재집계 플로우는 원래 한 번만 실행되어야 한다.

1. 대시보드 → 서버 `resync_requests`에 `consumed_at=NULL`로 저장
2. Agent 하트비트 → 서버가 응답에 `resync_since` 포함
3. Agent가 `sync_event_logs(max_events=500)` 실행 (wevtutil 2회: 일반 sync + resync)
4. Agent가 `ack_resync_to_server()` 호출 → 서버 `consumed_at = CURRENT_TIMESTAMP`
5. 다음 하트비트부터는 `consumed_at IS NOT NULL`이라 `resync_since` 미포함

그러나 `ack_resync_to_server()`는 **단발성 호출이고 실패 시 재시도가 없다**.
또한 Agent 측에 "이 요청은 이미 처리했다"는 **로컬 기록이 없다** — 유일한
진실 소스는 서버의 `consumed_at`. 따라서 네트워크 이슈·타임아웃 등으로 ack가
한 번이라도 실패하면:

- 서버 `consumed_at`은 영원히 NULL
- 매 하트비트(60초)마다 `resync_since`가 또 내려옴
- Agent가 매번 wevtutil 500개 조회 + 이벤트 재전송 반복
- 사용자는 매분 CMD 창을 봄 (결함 1과 결합되어 심각해짐)

결함 1만 고쳐도 창은 안 보이지만, 배경에서 매분 500개 이벤트 조회·전송
루프가 돌아 네트워크·CPU 낭비는 그대로 남는다. 따라서 결함 2도 반드시 해결.

## 수정 내용

### 1. `agent/installer.py` — `CREATE_NO_WINDOW` 플래그 추가

`get_boot_events_from_log()`와 `get_shutdown_events_from_log()`의
`subprocess.run` 호출에 `creationflags=0x08000000` 추가.

```python
result = subprocess.run(
    ['wevtutil', 'qe', 'System', ...],
    capture_output=True,
    text=True,
    timeout=30,
    creationflags=0x08000000,  # CREATE_NO_WINDOW
)
```

### 2. `agent/installer.py` — 재집계 처리 멱등성 도입

`send_heartbeat()` 내 재집계 처리부에 로컬 멱등성 체크를 추가.
`state.json`에 `last_processed_resync_since`를 기록하고, 동일 since가
또 내려오면 재집계는 스킵하고 ack만 재시도한다.

```python
resync_since_str = response_data.get('resync_since')
if resync_since_str:
    try:
        since_dt = datetime.fromisoformat(resync_since_str)
        state = load_state()
        last_processed = state.get('last_processed_resync_since')

        if last_processed == resync_since_str:
            # 이미 처리한 요청 — ack만 재시도
            log_error(f"[RESYNC] 이미 처리한 요청 재수신 (since={since_dt}), ack만 재시도")
            ack_resync_to_server(server_url)
        else:
            log_error(f"[RESYNC] 재집계 요청 수신: since={since_dt}")
            sent = sync_event_logs(server_url, since_override=since_dt, max_events=500)
            log_error(f"[RESYNC] 재집계 완료: {sent}개 전송")
            # ack 전에 로컬 기록 — ack 실패해도 재집계 반복 방지
            state['last_processed_resync_since'] = resync_since_str
            save_state(state)
            ack_resync_to_server(server_url)
    except Exception as e:
        log_error(f"[RESYNC] 재집계 실패 (다음 하트비트에 재시도): {e}")
```

**재집계 재요청 호환성**: 대시보드에서 동일 PC에 재집계를 또 요청하면
서버 `request_resync`가 `ON CONFLICT UPDATE`로 새 `since` 값을 덮어쓴다.
Agent에 저장된 `last_processed_resync_since`와 값이 다르므로 정상적으로
새 재집계가 실행된다.

### 3. `agent/version.py`

`AGENT_VERSION`을 `1.1.0` → `1.1.1`로 올림 (패치 버전).

## 검증 방법

1. `agent/build.bat`로 Agent EXE 재빌드, 테스트 PC에 배포
2. 대시보드에서 "재집계" 버튼 클릭 (7/14/30일)
3. **첫 번째 하트비트**: `agent.log`에 `[RESYNC] 재집계 요청 수신 → 재집계 완료` 기록, CMD 창 없음
4. **두 번째 하트비트 이후**: 서버에서 ack 처리됐다면 `resync_since` 더 이상 안 내려옴 (무소식)
5. **ack가 실패한 경우 시뮬레이션**: 서버 DB의 `consumed_at`을 일부러 NULL로 되돌려도
   다음 하트비트에 `[RESYNC] 이미 처리한 요청 재수신, ack만 재시도`가 기록되고
   wevtutil은 호출되지 않아야 함
6. 재집계를 다시 요청하면 새 `since` 값으로 정상 실행되는지 확인

## 호환성

- `creationflags=0x08000000`은 Windows 전용 상수 — Agent가 Windows 전용이므로 영향 없음
- 32비트 / 64비트 / Windows 7 빌드 모두 동일 플래그·로직 적용됨
- 이전 버전에서 설치된 PC는 `state.json`에 `last_processed_resync_since` 키가 없음 →
  `.get()`이 `None`을 반환 → 첫 재집계는 정상 실행 → 이후 기록됨. 마이그레이션 불필요

## 결함 3: 자동 복구 shutdown이 재집계 shutdown을 "중복"으로 블록

### 근본 원인

서버는 `_periodic_recovery_loop`(5분 주기, `server/main.py:205`)가
`check_and_recover_offline_shutdowns()`를 호출해, 하트비트가 끊긴 PC에
대해 **하트비트의 `last_seen` 값을 그대로 shutdown 이벤트로 INSERT** 한다
(`database.py:1026`). 이때 `event_record_id`는 NULL이고 timestamp는
실제 종료보다 보통 수 초~수 분 이른 근사값이다.

나중에 Agent가 재집계(또는 일반 sync)로 Windows 이벤트 로그에서
**실제 종료 시각과 `event_record_id`가 포함된 정확한 shutdown** 을
`send_event`로 전송하면, 서버 `insert_event`의 60초 중복 체크에 걸린다
(`database.py:193-203`):

- 중복 체크 1 (`event_record_id`): 자동 복구 이벤트는 record_id=NULL → 매칭 안 됨 → 통과
- 중복 체크 2 (60초 이내 같은 event_type): 매칭 → **중복 처리 후 저장 안 함**

결과: DB에는 부정확한 자동 복구 shutdown만 남고, 재집계로 가져온 정확한
shutdown은 버려진다. 대시보드 "마지막 종료"는 영원히 부정확한 근사값으로 고정.

### 수정 — `server/database.py` `insert_event`

60초 이내 동일 event_type이 이미 있을 때:

- **새 이벤트에 `event_record_id`가 있고** 기존 이벤트에 **`event_record_id`가 없으면**
  → 기존 행을 `UPDATE` 해서 정확한 timestamp·detail·source·record_id로 덮어쓴다
- 그 외(둘 다 record_id 있음, 또는 둘 다 없음)
  → 기존대로 중복 처리

이렇게 하면 "Windows 이벤트 로그에서 뽑은 권위 있는 원본"이 항상
근사값을 이긴다. 기존 자동 복구 이벤트도 record_id=NULL이므로 동일하게
덮어쓰기 대상이 되어, 마이그레이션 없이 기존 DB가 정상화된다.

### 부가 수정 — `check_and_recover_offline_shutdowns`

INSERT 시 `event_source='auto_recovery'`를 명시적으로 태깅
(`database.py:1026`). 덮어쓰기 조건은 `event_record_id IS NULL`만으로
충분하지만, 이후 대시보드/분석에서 자동 복구 이벤트를 구분할 수 있도록
메타데이터를 남긴다.

### 기존 데이터에 대한 영향

- 기존 DB에 저장된 자동 복구 shutdown은 `event_record_id=NULL` 상태라,
  다음에 Agent가 재집계나 일반 sync로 해당 시점의 실제 shutdown을 전송하면
  **자동으로 덮어써져 정확한 시각으로 갱신**된다
- 별도 SQL 마이그레이션 불필요
- Agent를 업데이트하지 않아도 **서버만 배포**하면 효과 발생
  (단, 재집계 반복/CMD 창 문제는 Agent 업데이트 필요)

### 검증 방법

1. 서버 재배포 후 문제 PC에서 대시보드 "재집계" 실행
2. Agent가 해당 날짜 종료 이벤트를 전송하면 서버 로그에 이벤트 수신 기록
3. 대시보드 새로고침 → "마지막 종료"가 실제 종료 시각(`18:35:12` 등)으로 갱신 확인
4. DB 확인 (선택):
   ```sql
   SELECT computer_name, event_type, timestamp, event_source, event_record_id
   FROM events
   WHERE computer_name = '<PC>'
   ORDER BY timestamp DESC LIMIT 10;
   ```
   재집계 이후 shutdown 행의 `event_record_id`가 NOT NULL이어야 함

## 결함 4: 자동 업데이트 배치 스크립트가 `find "<PID>"` 콘솔 창을 띄움

### 증상

자동 업데이트가 돌 때 화면에 제목이 `find "6544"`(숫자는 PID) 같은 검은
콘솔 창이 나타나 내용 없는 상태로 유지된다. 재집계와는 무관하며,
`--windowed`로 빌드된 Agent EXE 환경에서도 발생.

### 근본 원인

`agent/auto_updater.py:227` — `subprocess.Popen`이 배치 스크립트를
`creationflags=CREATE_NO_WINDOW | DETACHED_PROCESS`로 실행하고 있었다.
이 두 플래그는 MSDN상 **상호 배타적**이며 동시에 지정하면
`CreateProcess`가 기본 동작으로 폴백한다. 결과적으로 배치 스크립트의
자식들(특히 파이프라인 `tasklist | find "<PID>"`)이 새 콘솔을
할당받아 창이 뜬다.

더욱이 배치의 `wait_pid` 루프는 `goto wait_pid`로 PID가 살아있는 한
1초마다 반복 실행되어 창이 오래 유지된다.

### 수정 — `agent/auto_updater.py`

1. **Popen 플래그 정정** (`execute_update`):
   ```python
   CREATE_NO_WINDOW = 0x08000000
   startupinfo = subprocess.STARTUPINFO()
   startupinfo.dwFlags |= subprocess.STARTF_USESHOWWINDOW
   startupinfo.wShowWindow = subprocess.SW_HIDE
   subprocess.Popen(
       ['cmd', '/c', str(script_path)],
       creationflags=CREATE_NO_WINDOW,   # DETACHED_PROCESS 제거
       close_fds=True,
       startupinfo=startupinfo,          # 자식 창도 SW_HIDE
   )
   ```
2. **배치 스크립트의 파이프라인 제거** (`create_update_script`):
   `tasklist /FI "PID eq {pid}" | find "{pid}"` 루프를 제거하고
   `timeout /t 5` 단순 대기로 대체. 남은 프로세스는 뒤따르는
   `taskkill /F /IM <exe>`와 copy 재시도(5회)로 커버됨.

## 보너스 개선: 신규 Agent 설치 시 서버 재집계 루프 자동 해소

구버전 Agent에서 반복 중이던 재집계 요청은 `resync_requests.consumed_at`
이 서버 DB에 NULL로 남아있는 상태. 새 Agent를 설치해도 **첫 하트비트까지**
는 여전히 서버가 `resync_since`를 응답에 포함시킨다.

`agent/installer.py`의 `install_agent()`에서 `register_to_server` 직후
`ack_resync_to_server`를 즉시 호출하도록 추가했다. 실패하더라도 로컬
`state.json`에 `installed_at`을 기록해 두어, 다음 하트비트에서 멱등성
체크(결함 2의 수정)가 자연스럽게 루프를 차단한다.

실제 효과:
- Inno Setup 설치 직후 서버의 pending resync가 즉시 소비됨
- 재집계 요청을 사용자가 다시 누르지 않는 한 Agent는 재집계 스킵
- 루프 복구를 위해 수동 DB UPDATE를 실행할 필요 없음
